# 消息队列架构设计规范
## 一、架构总览
采用Redis Stream作为消息队列中间件，实现客户端与服务端完全解耦，服务端拆分为多个独立微服务模块，支持水平扩展。

```
┌─────────────────────────────────────────┐
│ 设备模拟器客户端                          │
│ 生成数据 → 本地缓存 → 发送MQ → 重传机制    │
└───────────────────┬─────────────────────┘
                    │
                    ▼
┌─────────────────────────────────────────┐
│ Redis Stream 消息队列                    │
│ ├─ upstream_data: 客户端上传原始数据队列  │
│ ├─ validated_data: 验证通过的数据队列    │
│ └─ dead_letter: 处理失败的死信队列        │
└───────────────────┬─────────────────────┘
                    │
        ┌───────────┴───────────┐
        ▼                       ▼
┌─────────────────┐     ┌─────────────────┐
│ 数据消费模块     │     │ 多副本消费       │
│ 从upstream读取   │     │ 负载均衡         │
└─────────┬───────┘     └─────────────────┘
          │
          ▼
┌─────────────────────────────────────────┐
│ 数据验证模块                              │
│ 格式校验/完整性校验/业务规则校验          │
│ 验证通过→validated_data队列               │
│ 验证失败→dead_letter队列                 │
└───────────────────┬─────────────────────┘
                    │
        ┌───────────┴───────────┐
        ▼                       ▼
┌─────────────────┐     ┌─────────────────┐
│ 数据写入模块     │     │ 日志记录模块     │
│ 写入文件/数据库   │     │ 记录操作日志     │
│ 生成报表         │     │ 统计监控         │
└─────────────────┘     └─────────────────┘
```

## 二、消息规范
### 2.1 消息格式（所有队列统一使用）
```json
{
  "msg_id": "uuid字符串",
  "device_id": "device_001",
  "timestamp": 1700000000000,
  "data": {
    "sensors": {"left_force_n": 300.0, "right_force_n": 280.0},
    "analysis": {"deviation_ratio": 0.034}
  },
  "retry_count": 0,
  "create_time": 1700000000
}
```
| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| msg_id | string | 是 | 全局唯一消息ID，UUID格式 |
| device_id | string | 是 | 设备ID |
| timestamp | number | 是 | 数据采集时间戳（毫秒） |
| data | object | 是 | 业务数据 |
| retry_count | number | 是 | 重试次数，首次发送为0 |
| create_time | number | 是 | 消息创建时间戳（秒） |

### 2.2 队列定义
| 队列名称 | 用途 | 消息保留时间 |
|----------|------|--------------|
| upstream_data | 客户端上传的原始数据队列 | 7天 |
| validated_data | 经过验证的合法数据队列 | 30天 |
| dead_letter | 处理失败的消息队列 | 永久 |

## 三、客户端重传机制
### 3.1 重传规则
1. 消息发送失败（超时/服务器错误）时，自动存入本地持久化缓存
2. 重传间隔采用指数退避：第1次重试间隔1s，第2次2s，第3次4s，最大间隔30s
3. 最大重试次数：10次，超过10次移入本地死信目录，人工处理
4. 客户端启动时自动加载本地缓存中未发送成功的消息，自动重传

### 3.2 本地缓存结构
```
client_cache/
├── pending/          # 待发送的消息（每条一个JSON文件）
├── dead_letter/      # 超过最大重试次数的消息
└── send_success/     # 发送成功的消息（可选保留，用于追溯）
```

## 四、服务端模块规范
所有模块都是独立进程，支持单独启停、多副本运行，通过Redis Stream消费者组实现负载均衡。

| 模块名称 | 职责 | 多副本支持 |
|----------|------|------------|
| consumer_worker | 从upstream_data队列消费数据，转发到验证流程 | 是，自动负载均衡 |
| validator_worker | 数据格式校验、业务规则校验 | 是 |
| writer_worker | 从validated_data队列消费数据，写入文件、生成报表 | 是 |
| logger_worker | 记录操作日志、统计指标、监控告警 | 否（单实例即可） |

### 4.1 消费者组配置
每个类型的工作节点使用同一个消费者组，Redis自动分配消息给不同副本，同一消息只会被一个副本消费。

## 五、可靠性保证
1. **至少一次投递**：所有消息必须收到ACK才会从队列删除，避免丢失
2. **幂等处理**：所有模块根据msg_id去重，重复消息不会重复处理
3. **死信队列**：处理失败的消息不会丢失，移入死信队列供人工排查
4. **持久化**：Redis开启AOF持久化，消息不会因为Redis重启丢失
